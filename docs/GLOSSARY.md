# Glossary

Alphabetical reference for every technical term used in the Zscaler RAG Incident Helper.
If a term isn't here, open a [GitHub Issue](https://github.com/manavendrayadav/zscaler-rag/issues).

---

## A

**BAAI/bge-m3**
An open-source text embedding model developed by the Beijing Academy of AI (BAAI). "bge" stands for "BERT-based Generative Embeddings", "m3" for Multi-lingual, Multi-functionality, Multi-granularity. It is the default embedding model in this project because it supports both dense and sparse vectors simultaneously (MTEB score: 64.3), making it significantly better than lighter models for retrieving exact error codes and Zscaler-specific terminology.
*Where it appears:* `config.py` (`EMBEDDING_MODEL`), `pipeline/embedder.py`

**Bearer token**
A type of HTTP authentication credential. When making API calls, you include it as: `Authorization: Bearer <your-key>`. The "Bearer" prefix is not optional — it is part of the HTTP authentication scheme. In this project, the default token is `zscaler-rag` (change before team deployment).
*Where it appears:* `api/main.py`, all `/v1/*` endpoints

---

## C

**Chunk**
A paragraph-sized segment of a Zscaler documentation page. Pages are too long to embed as a whole (embedding models have a ~512-token input limit), so each page is split into overlapping chunks of approximately 1,500 characters (~500 tokens). Chunks are the basic retrieval unit stored in Qdrant. The system currently holds ~13,800 chunks from 1,825 pages.
*Where it appears:* `pipeline/chunker.py`, `config.py` (`CHUNK_SIZE`, `CHUNK_OVERLAP`)

**Chunk ID**
A unique identifier for each chunk, generated as the MD5 hash of `url|section|chunk_index`. Because the ID is computed deterministically from the content's location, re-ingesting the same page always produces the same IDs — enabling safe "upsert" (update if exists) operations without creating duplicates.
*Where it appears:* `pipeline/chunker.py` line 103

**Chunk overlap**
The number of characters shared between consecutive chunks from the same document. A 150-character overlap (10% of 1,500 chars) prevents context from being lost at chunk boundaries. For example, if an important sentence spans two chunks, the overlap ensures the full sentence appears in at least one chunk.
*Where it appears:* `config.py` (`CHUNK_OVERLAP=150`)

**Collection**
Qdrant's equivalent of a database table. One collection named `zscaler_docs` holds all ~13,800 vector embeddings and their metadata (URL, product, section, chunk text). A collection must be created before vectors can be upserted.
*Where it appears:* `pipeline/indexer.py`, `config.py` (`COLLECTION_NAME`)

**Context window**
The maximum amount of text an LLM can process in one request, measured in tokens. Groq's llama-3.3-70b-versatile has a 128,000-token context window. The RAG system sends a system prompt + retrieved chunks + user query, all of which must fit within this limit.

**Cosine similarity**
A mathematical measure of similarity between two vectors, ranging from 0 (completely different) to 1 (identical). Qdrant uses cosine distance by default when comparing the query embedding to stored chunk embeddings. Scores above `MIN_SCORE` (default 0.3) are considered relevant.
*Where it appears:* `pipeline/indexer.py` (`Distance.COSINE`)

**Cross-encoder (re-ranker)**
A neural network model that takes a (query, document) pair as input and outputs a single relevance score. Used after the initial Qdrant search to re-rank the top-20 retrieved candidates by actual semantic similarity to the query. The model used is `cross-encoder/ms-marco-MiniLM-L-6-v2` (~90 MB). Cross-encoders are slower than bi-encoders (they process pairs, not individual texts) but significantly more accurate for ranking.
*Where it appears:* `rag/retriever.py`, baked into the Docker image

---

## D

**Dense vector**
A list of 1,024 floating-point numbers representing the semantic meaning of a text passage. Every word, phrase, and document has a unique dense vector. Texts with similar meaning have vectors that are close together in vector space (high cosine similarity). Dense vectors are generated by the bge-m3 model.
*Contrast with:* Sparse vector
*Where it appears:* `pipeline/embedder.py`, Qdrant collection `"dense"` index

**Docker profile**
A Docker Compose feature for grouping optional services. The Ollama service has the profile `local-llm` — it only starts when you pass `--profile local-llm` to docker-compose. This keeps the default `make up` fast by not pulling or starting a large GPU-capable container unless explicitly requested.
*Where it appears:* `docker-compose.yml` (`profiles: [local-llm]`)

**Drain3**
A Python library for mining log templates from raw log text. Given many log lines, Drain3 clusters them by template (e.g., `ERROR connector=<*> reason=<*>`) and extracts structured parameters. The system uses Drain3 in log-analysis mode to extract searchable signals (error codes, component names) from pasted log dumps.
*Where it appears:* `rag/log_parser.py`, `requirements.txt`

---

## E

**Embedding**
The process of converting a piece of text into a dense vector (a list of numbers). The resulting vector captures the semantic meaning of the text, allowing mathematically similar texts to be found even if they don't share keywords. The bge-m3 model generates both dense (1024-d) and sparse embeddings simultaneously.
*Where it appears:* `pipeline/embedder.py`

**Embedding cache**
A binary `.npz` file (`data/embeddings_cache.npz`) that stores all computed embeddings after the 4-hour bge-m3 computation completes. If the subsequent Qdrant upsert fails, the embeddings can be reloaded from cache — skipping the expensive recomputation — by running `python scripts/ingest.py --skip-embed`.
*Where it appears:* `scripts/ingest.py`

---

## H

**Hybrid search**
A retrieval strategy that combines dense semantic search and sparse keyword search, then merges the two ranked result lists using Reciprocal Rank Fusion (RRF). Hybrid search is more effective than either alone: dense search finds semantically similar passages even without shared keywords, while sparse search ensures exact error codes (e.g., `AUTH_FAILED`) are always found.
*Where it appears:* `rag/retriever.py` (`_search_hybrid`)

---

## I

**Ingest**
The full pipeline that processes raw crawled Markdown files into searchable vector embeddings stored in Qdrant: Load `.md` files → chunk → embed with bge-m3 → upsert to Qdrant → update manifest. Run with `make ingest`.
*Where it appears:* `scripts/ingest.py`

---

## L

**LLM (Large Language Model)**
An AI model trained on large text corpora that can generate, summarise, and reason about text. In this project, the LLM is the final step: it receives the user's question + retrieved documentation chunks and generates a structured resolution. The LLM itself knows nothing about your specific Qdrant data — it relies entirely on the retrieved context.
*Providers in this project:* Groq, OpenRouter, Ollama

**Log analysis mode**
An automatic query mode triggered when the user pastes multi-line text containing timestamps and error keywords (e.g., `2026-05-28T10:00:00Z ERROR connector=prod-dc1 reason=AUTH_FAILED`). In this mode, Drain3 extracts structured signals, which are used to retrieve relevant documentation, and the LLM is prompted to produce an "Identified Events + Root Cause + Resolution" response instead of the standard doc_search format.
*Where it appears:* `api/main.py`, `rag/log_parser.py`

---

## M

**Manifest file** (`data/crawl_manifest.json`)
A JSON file that tracks the crawl state for every Zscaler documentation URL: last crawl timestamp, sitemap `lastmod` date, and which Qdrant chunk IDs belong to each page. Used by `make update` to skip unchanged pages and by the ingest pipeline to maintain chunk-to-page mapping. Gitignored — regenerated by a fresh crawl.

**MIN_SCORE**
The minimum cosine similarity score (0–1) for a retrieved chunk to be included in the LLM context. The default is `0.3`. Chunks scoring below this threshold are discarded before generation, preventing low-quality matches from degrading response accuracy. Tune lower (e.g. `0.1`) for broader recall, higher (e.g. `0.5`) for higher precision.
*Where it appears:* `config.py`, `rag/retriever.py`

---

## O

**Ollama**
An open-source runtime for running LLM models locally on your own hardware. When using Ollama, no data leaves your machine. Required for processing internal Zscaler logs, screenshots containing sensitive information, or any query involving employee/system identifiers.
*Where it appears:* `llm/ollama_provider.py`, `docker-compose.yml` (profile: `local-llm`)

**OpenAI-compatible API**
An HTTP API that accepts the same request and response format as OpenAI's Chat Completions API (`POST /v1/chat/completions`, `GET /v1/models`). The RAG API exposes this interface, which means any tool built for OpenAI (including OpenWebUI) connects to it without modification.
*Where it appears:* `api/main.py`

---

## P

**Product filter**
An optional parameter (`product_filter`) in the chat completions API that restricts Qdrant retrieval to chunks from one Zscaler product. Valid values: `"zia"`, `"zpa"`, `"zdx"`, `"deception"`. Improves precision when you know which product is causing the incident. Can also be set via the `[product:zpa]` prefix in chat messages.
*Where it appears:* `api/models.py`, `rag/retriever.py`

---

## Q

**Qdrant**
An open-source vector database written in Rust, optimised for semantic similarity search. In this project, it stores ~13,800 chunk embeddings with metadata (URL, product, section, text). Runs as a Docker container, exposes HTTP on port 6333 and gRPC on port 6334, and persists data in a Docker volume (`qdrant_storage`).
*Homepage:* https://qdrant.tech
*Where it appears:* `docker-compose.yml`, `pipeline/indexer.py`, `rag/retriever.py`

---

## R

**RAG (Retrieval-Augmented Generation)**
A technique for improving LLM accuracy by retrieving relevant documents before generating an answer. Instead of relying on the LLM's training data (which may be stale or incomplete), RAG fetches relevant passages from a curated knowledge base and includes them in the prompt. This "grounds" the response in actual documentation, dramatically reducing hallucination.
*Analogy:* Open-book exam vs. closed-book. RAG is open-book — the AI reads the relevant pages before answering.

**Re-ranking**
The process of using a cross-encoder model to re-score the top-N candidates retrieved from Qdrant, producing a more accurate ranking than the initial vector search. The system fetches up to 20 candidates from Qdrant, then the cross-encoder scores each (query, chunk) pair, and the top-K highest-scoring chunks are passed to the LLM.
*Where it appears:* `rag/retriever.py` (`_get_reranker`)

**RRF (Reciprocal Rank Fusion)**
A method for combining two ranked result lists (dense + sparse) into a single merged ranking. Each item's combined score is the sum of `1/(k + rank)` across both lists (where `k` is a constant, typically 60). RRF is preferred over simple score averaging because it handles different score scales gracefully.
*Where it appears:* `rag/retriever.py` (`FusionQuery(fusion=Fusion.RRF)`)

---

## S

**Sentence-transformers**
A Python library providing pre-trained transformer models for generating text embeddings. Used as the embedding backend when `EMBEDDING_MODEL` is not bge-m3 (e.g., `all-MiniLM-L6-v2`). Also used for the cross-encoder model (`CrossEncoder` class).
*Where it appears:* `pipeline/embedder.py`, `rag/retriever.py`

**Sparse vector**
A dictionary mapping token IDs to their importance weights (BM25-style). Unlike dense vectors which represent holistic meaning, sparse vectors capture exact keyword presence — making them ideal for finding specific error codes, product names, and CLI commands. Combined with dense vectors in hybrid search.
*Contrast with:* Dense vector
*Where it appears:* `pipeline/embedder.py`, Qdrant collection `"sparse"` index

---

## T

**TOP_K**
The number of chunks retrieved from Qdrant per query (default: 5). After retrieval and re-ranking, the top-K chunks (by cross-encoder score, above MIN_SCORE) are included in the LLM prompt as context. Increasing this value provides richer context at the cost of slower queries and higher risk of the LLM being distracted by marginal matches.
*Where it appears:* `config.py`, API request field `top_k`

**Token**
The basic unit of text for LLMs — roughly 0.75 words or 4 characters. Pricing for cloud LLMs is per token. A 1,500-character chunk is approximately 375–500 tokens. The default `max_tokens=2048` limits the length of the generated response.

---

## U

**Upsert**
A database operation that inserts a new record if it doesn't exist, or updates the existing record if it does (based on the chunk ID). Because chunk IDs are deterministic (MD5 hash), re-ingesting a page simply overwrites its existing Qdrant points — no duplicate data is created.
*Where it appears:* `pipeline/indexer.py`

---

## V

**Vector**
A list of numbers representing a piece of text. Dense vectors have 1,024 numbers (for bge-m3); sparse vectors are dictionaries of token ID → weight pairs. Vectors that are mathematically "close" represent semantically similar text.

**Vector database**
A database optimised for storing and searching vectors. Unlike relational databases (which filter rows by exact value), vector databases rank results by similarity. Qdrant is the vector database used in this project.

---

## Z

**ZCC (Zscaler Client Connector)**
The endpoint agent installed on user devices that routes traffic through Zscaler's cloud. Relevant to this project as a source of log data for incident investigation.

**ZDX (Zscaler Digital Experience)**
Zscaler's digital experience monitoring product. Tracks end-user application performance, latency, and experience scores.

**ZIA (Zscaler Internet Access)**
Zscaler's secure web gateway product. Handles internet-bound traffic, SSL inspection, policy enforcement, and threat prevention.

**ZPA (Zscaler Private Access)**
Zscaler's zero-trust network access product. Provides secure access to internal applications without a VPN. Uses App Connectors and Brokers.
